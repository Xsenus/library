# Полное объяснение: что происходит в программе

Этот документ описывает «жизнь» приложения от входа пользователя до сложных AI‑сценариев.

## 1. Запуск и инициализация

1. Next.js поднимает приложение и роутер App Router.
2. Для серверной части сразу доступны API‑роуты (`app/api/**`) и server components.
3. Подключения к БД создаются лениво через `lib/db.ts` и `lib/db-bitrix.ts`.
4. На клиенте рендерятся страницы (`app/**`) и интерактивные компоненты (`components/**`).

## 2. Контур безопасности

### 2.1 Middleware

`app/middleware.ts` проверяет каждый запрос к страницам:

- если путь публичный (`/login`, `/_next/*`, `/embed/*` и т.д.) — пропускает;
- если это API (`/api/*`) — тоже пропускает (авторизация решается в самих API);
- иначе требует cookie `cin_session`.

При отсутствии cookie делает редирект на `/login?next=...`.

### 2.2 Protected layout

Даже если cookie есть, `app/(protected)/layout.tsx` делает **живую** проверку пользователя:

- читает сессию;
- запрашивает актуальный статус пользователя в БД (`getLiveUserState`);
- если пользователь деактивирован — редирект на логин.

Это защищает от ситуации «старый токен, но доступ уже отозван».

## 3. Авторизация

### 3.1 Вход

`POST /api/auth/login`:

1. Валидирует тело (`login`, `password`, `remember`).
2. Ищет пользователя в `users_irbis`.
3. Сверяет пароль:
   - bcrypt‑хэш, если поле похоже на bcrypt;
   - plain compare для legacy‑учёток.
4. Проверяет `activated`.
5. Создаёт JWT‑сессию (`cin_session`) через `lib/auth.ts`.

### 3.2 Проверка «кто я»

`GET /api/auth/me`:

- расшифровывает JWT;
- перепроверяет `activated`/`irbis_worker` в БД;
- подтягивает логин, лимиты и вычисляет флаги `is_admin` и `unlimited`.

UI регулярно использует этот endpoint для актуализации прав.

### 3.3 Выход

`POST /api/auth/logout` очищает cookie сессии.

## 4. Основной пользовательский сценарий (Library)

Главный экран — `app/(protected)/library/LibraryClient.tsx`.

### 4.1 Состояние в клиенте

Компонент держит:

- активный таб (`library`, `cleanscore`, `okved`, `aisearch`, `aianalysis`, `aidebug`);
- профиль пользователя (`isWorker`, `isAdmin`);
- выбранные узлы иерархии;
- поисковые строки с debounce;
- отдельные состояния таблиц (CleanScore, OKVED, и т.д.);
- состояние квоты просмотров.

### 4.2 Иерархия данных

Запросы идут по цепочке API:

1. `/api/industries`
2. `/api/industries/[industryId]/prodclasses`
3. `/api/prodclasses/[prodclassId]/workshops`
4. `/api/workshops/[workshopId]/equipment`
5. `/api/equipment/[id]` (детальная карточка)

Все endpoint’ы:

- валидируют query/params через Zod;
- используют SQL с pagination;
- нормализуют ответ под общие `ListResponse` схемы.

## 5. Карточка оборудования и дневные лимиты

Когда пользователь открывает оборудование (`/api/equipment/[id]`), backend:

1. проверяет сессию и живой статус;
2. загружает карточку (чтобы корректно вернуть 404 независимо от квоты);
3. вычисляет лимит через `resolveUserLimit`;
4. учитывает просмотр в `users_activity` как уникальный за день;
5. возвращает карточку + заголовки остатка квоты.

Если лимит исчерпан — возвращает 403 с понятной ошибкой.

## 6. CleanScore / OKVED / B24

### CleanScore

Отдельные API дают сводные строки по оборудованию, рейтингам и фильтрам. В UI можно подтверждать `equipment_score_real`.

### OKVED

Сценарий опирается на endpoint’ы `/api/okved/*`: выборка кодов, компаний и связок с оборудованием.

### Bitrix24

`/api/b24/*` работает как адаптер к webhook Bitrix:

- резолв компании по ИНН;
- ответственные;
- контакты;
- мета‑данные и цветовые статусы.

Часть запросов кэшируется/нормализуется, чтобы не перегружать внешний API.

## 7. AI Search

`/api/ai-search` — гибридный поиск:

1. Быстрый локальный SQL поиск по нескольким сущностям.
2. Если нужно — обращение к внешнему AI‑сервису.
3. При наличии ключа — использование OpenAI embeddings.
4. Нормализация результатов в единый payload (`goods`, `equipment`, `prodclasses`).
5. Логирование деталей в `ai_debug_events`.

Это позволяет держать хороший UX: мгновенный ответ от SQL + более умная догрузка от AI.

## 8. AI Analysis (очередь фонового анализа)

Это самый сложный участок.

### 8.1 Запуск

`POST /api/ai-analysis/run`:

- принимает список ИНН и режим (`steps`/`full`);
- проверяет доступность AI интеграции (`/health`);
- пишет задания в `ai_analysis_queue`;
- обновляет статусы в данных компаний;
- запускает обработчик очереди.

### 8.2 Обработчик очереди

Worker:

- берёт advisory lock (чтобы одновременно работал только один экземпляр);
- читает старую задачу `FOR UPDATE SKIP LOCKED`;
- проверяет stop‑команды;
- переводит компанию в `running`;
- выполняет шаги AI‑пайплайна;
- пишет прогресс/ошибки/финал.

### 8.3 Остановка

`POST /api/ai-analysis/stop` ставит локальную команду stop. Следующая итерация воркера увидит её и не продолжит обработку.

### 8.4 Наблюдаемость

`/api/ai-debug/events` хранит события трёх классов:

- traffic,
- error,
- notification.

UI‑вкладка AI Debug показывает фильтруемый журнал и помогает разбирать сбои интеграции.

## 9. Изображения и embed‑режим

### 9.1 Картинки

- `/api/images/google` оборачивает Google CSE.
- `/api/images/proxy` используется как прокси, если нужно безопасно отдать удалённое изображение.

### 9.2 Embed

`/embed/equipment`:

- публичный маршрут без логина;
- ищет запись по `hash_equipment`;
- показывает компактную карточку для iframe.

`next.config.js` добавляет заголовки, разрешающие внешнее встраивание только для `/embed/*`.

## 10. Почему система устроена так

1. **Next.js как единый full‑stack** уменьшает стоимость интеграции UI↔API.
2. **PostgreSQL как центр данных** упрощает согласованность и SQL‑аналитику.
3. **Отдельный пул Bitrix БД** снижает риск блокировок и изоляционно работает с внешней схемой.
4. **Очередь AI в БД** даёт предсказуемость и простой контроль состояния без отдельного брокера.
5. **Zod‑валидация в API** режет класс ошибок на границе входа.
6. **Debug‑журнал AI** повышает наблюдаемость и ускоряет разбор продовых проблем.
