# План Исправлений И Проверок Связки AI Анализа Компаний

Дата ревизии: 2026-03-30

Контур:
- `library`
- `ai-integration`
- `ai-site-analyzer`

Этот документ фиксирует актуальный план исправлений и проверок после повторной синхронизации репозиториев. Он нужен как рабочий backlog для правок, тестов и поэтапного вывода в стабильное состояние.

## Что Уже Исправлено После Синхронизации

- В `ai-integration` `/health` теперь корректно учитывает отключенные DSN и не валит `ok`, если подключение не настроено.
- В `ai-integration` схема `analyze_json` уже содержит `okved_fallback_used`, поэтому старое замечание про отсутствие поля больше неактуально.
- В `ai-integration` fallback по ОКВЭД и возврат признаков `site_unavailable`/`okved_fallback_used` уже присутствуют в runtime-логике.

## Актуальные Проблемы, Которые Надо Закрыть

### P0

- `library` считает `/v1/pipeline/full` успешным по самому факту HTTP `200`, не проверяя `errors` в теле ответа.
- Из-за этого во вкладке и в очереди можно увидеть ложный статус завершения анализа.

### P1

- `library` и `ai-integration` по-разному трактуют `/health`: UI-хелпер считает любой HTTP `200` доступностью, а раннер смотрит на `body.ok`.
- В `library` смешаны env-переменные адреса upstream и downstream: `AI_INTEGRATION_*` и `AI_ANALYZE_*`/`ANALYZE_BASE`.
- Fallback-проверки в `library` повторно запускают тяжелые шаги через `POST /v1/analyze-json` и `POST /v1/parse-site`, хотя используются как probe.
- Fallback для `ib_match` берет `client_id` из `dbBitrix`, хотя основная выдача вкладки работает с `db`.

### P2

- Недостаточно явный машинный контракт между тремя сервисами по статусам: `ok`, `partial`, `failed`, `fallback`, `site_unavailable`, `retryable`.
- Недостаточно тестов на регрессии уровня "межсервисный поток", особенно на частичные ошибки и fallback-сценарии.

## Общий План Работ

### Этап 1. Зафиксировать Контракты Между Сервисами

Цель: перестать полагаться на неявные договоренности и сделать статусы машинно-проверяемыми.

Что делаем:
- В `ai-integration` расширяем контракт `/v1/pipeline/full`, чтобы ответ содержал явный итог:
  - `ok: boolean`
  - `status: "completed" | "partial" | "failed"`
  - `failed_steps: string[]`
  - `errors: PipelineStepError[]`
- В `library` используем именно эти поля, а не только HTTP-статус.
- Для `parse-site` и `analyze-json` отдельно фиксируем, какие ответы считаются:
  - штатным успехом
  - fallback-успехом
  - частичной ошибкой
  - ошибкой, требующей ретрая
- В `ai-site-analyzer` и `ai-integration` фиксируем список машинных причин отказа:
  - `site_unavailable`
  - `empty_site_content`
  - `external_service_unavailable`
  - `llm_parse_error`
  - `validation_error`

Результат этапа:
- `library` больше не может "придумать" успешный статус поверх частично упавшего пайплайна.
- Появляется единый словарь статусов для всех трех сервисов.

### Этап 2. Починить Правду О Состоянии Пайплайна

Цель: статусы в UI, очереди и логах должны совпадать с фактическим исходом.

Что делаем:
- В `library` меняем обработчик полного пайплайна:
  - успех только если `res.ok === true` и `payload.errors.length === 0`, либо `payload.ok === true`
  - частичный успех отдельно сохраняется и отдельно показывается
  - полный провал не должен помечаться как `completed`
- В таблицах/очереди добавляем или переиспользуем явный статус:
  - `completed`
  - `partial`
  - `failed`
  - `queued`
  - `running`
- Логи `safeLog` должны фиксировать не только transport success, но и domain success.

Проверки:
- Один шаг внутри `/v1/pipeline/full` падает, HTTP остается `200`, а UI и БД показывают `partial` или `failed`, но не `completed`.
- `errors` из `ai-integration` сохраняются и видны в отладке.

### Этап 3. Выровнять Контракт `/health`

Цель: UI и серверный раннер должны одинаково понимать, доступен сервис или нет.

Что делаем:
- В `library/lib/ai-integration.ts` читаем из `/health` не только HTTP-статус, но и `body.ok`.
- Возвращаем из хелпера расширенный результат:
  - `available`
  - `ok`
  - `connections`
  - `detail`
- В UI вкладки показываем:
  - "сервис недоступен"
  - "сервис доступен частично"
  - "сервис готов к запуску"
- В `ai-integration` документируем, что контракт `/health` опирается на включенные DSN и всегда возвращает `ok` и `connections`.

Проверки:
- HTTP `200` + `ok: false` отображается как проблема и в UI, и в раннере.
- Отключенная необязательная БД дает `disabled`, а не ложное падение всего health-check.

### Этап 4. Развести Env-Переменные По Уровням Ответственности

Цель: `library` должен знать адрес только `ai-integration`, а `ai-integration` должен знать адрес только своего downstream-сервиса.

Что делаем:
- В `library` оставляем для upstream только:
  - `AI_INTEGRATION_BASE_URL`
  - `AI_INTEGRATION_BASE`
- Убираем fallback на:
  - `AI_ANALYZE_BASE`
  - `ANALYZE_BASE`
- В `ai-integration` оставляем `AI_ANALYZE_BASE`/`ANALYZE_BASE` только как адрес внешнего analyze-сервиса.
- Обновляем `.env.example`, README и runtime-ошибки, чтобы по названию было очевидно, какой URL за что отвечает.

Проверки:
- Если задан только `ANALYZE_BASE`, `library` не должен пытаться ходить в downstream.
- При неправильной настройке `library` должен падать с понятной конфигурационной ошибкой, а не тихо ходить не туда.

### Этап 5. Сделать Fallback-Проверки Без Побочных Эффектов

Цель: probe не должен повторно запускать тяжелую обработку.

Что делаем:
- В `library` убираем схему "сначала GET, потом POST как проверка fallback".
- В `ai-integration` добавляем один из вариантов:
  - отдельный read-only endpoint с последним статусом анализа по ИНН
  - либо расширяем текущие GET-эндпоинты так, чтобы они никогда не инициировали перерасчет
- Для `parse-site` и `analyze-json` явно разделяем:
  - `GET status/state`
  - `POST run/recompute`
- Fallback по ОКВЭД должен определяться из уже сохраненного состояния, а не через повторный вызов вычисления.

Проверки:
- Повторный просмотр статуса по ИНН не порождает новых записей, новых запросов к внешнему сервису и новых тяжелых job.
- Количество вызовов downstream при неуспешном `ib_match` не удваивается.

### Этап 6. Привести Источник `client_id` К Одной Модели Данных

Цель: `clients_requests` должен читаться из понятного, единого источника, либо должна быть явно оформлена схема fallback между БД.

Что делаем:
- Проверяем, где `clients_requests` является канонической таблицей для AI-вкладки.
- В `library` переносим `getClientRequestIdByInn()` на ту же БД, из которой строится выдача вкладки.
- Если нужны обе БД, делаем явный порядок поиска:
  - primary source
  - fallback source
  - лог причины выбора fallback
- Документируем, какие таблицы считаются master-copy для:
  - `clients_requests`
  - `pars_site`
  - `dadata_result`
  - `ai_site_*`

Проверки:
- По одному и тому же ИНН `client_id` находится тем же способом, каким строится карточка компании в UI.
- Fallback для `ib_match` не отваливается только из-за того, что запрос ушел в другую БД.

### Этап 7. Усилить Набор Тестов И Проверок

Цель: повторная поломка контрактов должна ловиться до релиза.

Что делаем:
- Добавляем unit-тесты на парсинг контрактов в `library`.
- Добавляем API-тесты в `ai-integration` на:
  - `pipeline/full` с частичной ошибкой
  - `/health` с `disabled` DSN
  - `analyze-json` fallback по ОКВЭД
  - `parse-site` fallback без сайта
- Добавляем downstream-тесты в `ai-site-analyzer` на:
  - валидный success
  - пустой вход
  - машинно-классифицируемую ошибку
  - стабильность схем ответа
- Собираем ручной end-to-end чеклист на один ИНН для каждого ключевого сценария.

## План По Сервисам

## `library`

### Что Исправлять В Коде

- `lib/ai-integration.ts`
  - убрать чтение downstream env-переменных
  - сделать корректный health-parser
  - добавить типизацию тела `/health`
- `app/api/ai-analysis/run/route.ts`
  - проверять `errors` и/или `payload.ok` у `/v1/pipeline/full`
  - разделить transport success и business success
  - убрать side-effect probe для fallback
  - привести `getClientRequestIdByInn()` к канонической БД
- при необходимости `components/library/ai-company-analysis-tab.tsx`
  - показать пользователю `partial` и `health degraded`

### Проверки Для `library`

- Юнит-тесты:
  - health с `HTTP 200 + ok:false`
  - pipeline full с `errors.length > 0`
  - корректный выбор базового URL
- Интеграционные проверки:
  - частично успешный пайплайн
  - неуспешный health
  - fallback без повторного запуска analyze-step
- Ручная проверка:
  - UI не показывает ложный `completed`
  - статус в очереди и фактический результат совпадают

## Общий План Проверок После Исправлений

### Контрактные Сценарии

Нужно прогнать минимум такие сценарии:

1. Полностью успешный ИНН с валидным сайтом.
2. ИНН без доступного сайта, но с успешным fallback по ОКВЭД.
3. ИНН, у которого `parse-site` успешен, а `analyze-json` падает.
4. ИНН, у которого `pipeline/full` возвращает `errors`, но не транспортную ошибку.
5. Health c `HTTP 200`, но `ok: false`.
6. Конфигурация, где задан только `AI_INTEGRATION_BASE`.
7. Ошибочная конфигурация, где задан только `ANALYZE_BASE`.
8. Повторный просмотр статуса по ИНН без повторного расчета.
9. `ib_match` через `by-inn`, когда `client_id` находится в основной БД.
10. `ib_match` через fallback-источник, если это действительно поддерживаемый сценарий.

### Наблюдаемость

После исправлений нужно обязательно видеть в логах:
- исходный `request_id`
- ИНН
- имя шага
- transport status
- business status
- тип ошибки
- использован ли fallback
- был ли повторный перерасчет

## Порядок Внедрения

1. Сначала правим контракт `pipeline/full` и его обработку в `library`.
2. Потом выравниваем `/health`.
3. Потом чистим env-конфигурацию.
4. Потом убираем side-effect fallback probes.
5. Потом добиваем источник `client_id` и модель данных.
6. После этого прогоняем контрактные тесты и ручной end-to-end.
7. Только затем обновляем продовые `.env` и включаем новые правила в реальном запуске.

## Definition Of Done

Работа считается завершенной, когда одновременно выполнены все условия:

- нет ложных `completed` для частично упавшего пайплайна
- UI и сервер одинаково трактуют `/health`
- `library` не может случайно пойти напрямую в `ai-site-analyzer`
- fallback-проверки не запускают новые тяжелые расчеты
- `client_id` находится из понятного и документированного источника
- есть автоматические тесты на частичные ошибки и fallback
- ручной end-to-end чеклист пройден на актуальном стенде
